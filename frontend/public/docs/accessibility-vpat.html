VPAT 2.4 — WCAG 2.1 Edition

Accessibility Conformance Report

Voluntary Product Accessibility Template for ON24 Nexus — Conversational Analytics Platform

Product: ON24 Nexus v1.0 Date: March 14, 2026 Standard: WCAG 2.1 Level AA Contact: accessibility@on24.com

Product Description

ON24 Nexus is a web-based conversational analytics platform that enables users to explore ON24 webinar data through AI-powered chat, interactive charts, event calendars, and content management. The application runs in modern web browsers and uses WebSocket for real-time communication.

Evaluation Methods

  • Manual code review of all React components for semantic HTML and ARIA attributes
  • Automated test suite (35 accessibility contract tests — 100% pass)
  • Keyboard navigation testing across all interactive components
  • Color contrast analysis of CSS variables in light and dark themes
  • Screen reader compatibility review (structure, landmarks, live regions)

WCAG 2.1 Level A — Conformance

CriterionNameConformanceRemarks
1.1.1Non-text Content Supports All SVG icons use aria-hidden="true" with adjacent text labels. Chart components have descriptive titles. Calendar icon button has aria-label.
1.2.1Audio-only and Video-only N/A Application does not include audio or video content.
1.3.1Info and Relationships Supports Header uses <header>, sidebar uses <aside>, main content uses <main>, breadcrumb uses <nav aria-label>. Chat message list uses role="log" with aria-live="polite". Collapsible sections use aria-expanded.
1.3.2Meaningful Sequence Supports DOM order matches visual order: header → sidebar → main → chat input.
1.3.3Sensory Characteristics Supports Instructions do not rely solely on shape, size, or color. Connection status uses both color dot and text label.
1.4.1Use of Color Supports Agent routing uses colored borders AND text labels. PROD/QA badge uses color AND text. Pass/fail tags use color AND text.
1.4.2Audio Control N/A No audio playback.
2.1.1Keyboard Partially Supports All buttons and links are keyboard-focusable. Chat input handles Enter to submit. Calendar tooltips dismiss on Escape key. Gap: Calendar date grid not fully keyboard-navigable with arrow keys.
2.1.2No Keyboard Trap Supports Modal calendar can be closed with close button. No keyboard traps detected.
2.2.1Timing Adjustable Supports No time limits on user actions. WebSocket reconnects automatically.
2.3.1Three Flashes Supports No flashing content. Transitions use CSS with duration ≤200ms.
2.4.1Bypass Blocks Supports Landmark regions (<header>, <aside>, <main>) allow skip via screen reader. "Skip to main content" link appears on keyboard focus.
2.4.2Page Titled Supports HTML title set to "ON24 Nexus". Documentation pages have descriptive titles.
2.4.3Focus Order Supports Tab order follows visual layout: header → sidebar → main content → chat input.
2.4.4Link Purpose (In Context) Supports Document links have descriptive labels. External links have ↗ indicator.
3.1.1Language of Page Supports <html lang="en"> set on all pages.
3.2.1On Focus Supports No context changes on focus. Hover effects are decorative only.
3.2.2On Input Supports Chat sends on explicit Enter key or button click. No auto-submit.
3.3.1Error Identification Partially Supports API errors show "An error occurred" in chat. Gap: Validation errors (empty message, length limit) shown as text but not linked to input field via aria-describedby.
3.3.2Labels or Instructions Supports Chat input has visually hidden <label> element linked via htmlFor, plus aria-label attribute. Placeholder text provides additional context.
4.1.1Parsing Supports React generates valid HTML. No duplicate IDs detected.
4.1.2Name, Role, Value Supports Interactive elements have appropriate roles. Dark mode toggle has aria-pressed. Calendar button has aria-label. User menu button has aria-label.

WCAG 2.1 Level AA — Conformance

CriterionNameConformanceRemarks
1.3.4Orientation Supports No orientation lock. Layout adapts to viewport.
1.3.5Identify Input Purpose Partially Supports Gap: Chat input does not have autocomplete attribute. Not critical for chat but required for form fields.
1.4.3Contrast (Minimum) Supports Light mode: Text (#1a1d2e on #f8f9fc) = 14.2:1 — passes. Secondary text (#718096 on #fff) = 4.5:1 — passes. Dark mode: Text (#c9cfe0 on #0f1117) = 10.8:1 — passes. Secondary text (#94a0b8 on #0f1117) = 5.5:1 — passes.
1.4.4Resize Text Supports Uses rem/em units. Content reflows at 200% zoom. Sidebar collapses gracefully.
1.4.5Images of Text Supports No images of text. All text is rendered as HTML. SVG diagrams in docs use real <text> elements.
1.4.10Reflow Partially Supports Main layout reflows. Gap: Calendar modal at 96vw may not reflow well at 320px viewport width. Data tables may require horizontal scroll.
1.4.11Non-text Contrast Partially Supports Connection dot (green/red) has adjacent text. Gap: Chart axes, grid lines, and some UI borders may fall below 3:1 against background.
1.4.12Text Spacing Supports No content loss when text spacing is increased per WCAG requirements (line-height 1.5×, spacing 0.12em, word spacing 0.16em, paragraph spacing 2×).
1.4.13Content on Hover or Focus Supports Tooltips on calendar events use custom positioned divs. Escape key dismisses hover content. Tooltips persist until user action.
2.4.5Multiple Ways Supports Content reachable via: sidebar navigation, chat commands, suggestion tiles, calendar, breadcrumb hierarchy.
2.4.6Headings and Labels Partially Supports Doc pages use proper heading hierarchy. Gap: Main chat interface has no visible headings — relies on sidebar label and landmark structure.
2.4.7Focus Visible Supports Global :focus-visible rule provides 2px solid outline on all interactive elements (buttons, links, [tabindex]). Non-keyboard focus suppressed via :focus:not(:focus-visible).
3.1.2Language of Parts Supports Application is English-only. No mixed-language content.
3.2.3Consistent Navigation Supports TopNav and sidebar are consistent across all pages. Same position, same order.
3.2.4Consistent Identification Supports Same icons and labels used throughout (calendar, dark mode, chat). Consistent chip styling per agent type.
3.3.3Error Suggestion Partially Supports Empty message validation returns "Message cannot be empty." Gap: No auto-correction suggestions. Error messages could be more helpful.
3.3.4Error Prevention Supports Admin write operations (create event, add registrant) require explicit confirmation before execution.
4.1.3Status Messages Supports Agent routing uses aria-live="polite". Chat messages in role="log" container. Loading states have role="status" with aria-label. Processing indicator announced to screen readers.
