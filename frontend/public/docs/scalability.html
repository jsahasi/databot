3. Critical Bottlenecks & Mitigations

3.1 Anthropic API Rate Limits Critical

Each user message triggers 2–4 Anthropic API calls (orchestrator classification + sub-agent tool loop). At 6K concurrent, this is ~160 calls/second.

  • Prompt caching: Add cache_control to system prompts — 90% input cost reduction, minimal code change
  • Response caching: Redis cache for identical queries within 5 min window
  • Model tiering: Haiku for orchestrator routing (faster, cheaper), Sonnet for sub-agents
  • Rate limit increase: Request Anthropic production-tier rate limits

3.2 ON24 Master DB Connection Pool High

External read-only DB shared across all workers. Current config: 3 connections per worker × 5 workers = 15 total.

  • Current (dev): 15 connections — adequate for ~25-50 concurrent users
  • At 2K: Increase to 5/worker × 10 workers = 50 connections, or add PgBouncer
  • At 6K: PgBouncer as connection multiplexer is required (multiplex 200+ app connections → 50 DB connections)
  • Query caching: Redis cache for hot queries (event lists, KPIs) with 60s TTL

3.3 WebSocket Connection Management High

WebSocket connections are inherently sticky to a single worker (TCP). In-memory conversation history works without shared state.

  • Current (dev): 5 workers handle ~10 WS connections each — no issues
  • At 2K: ~400 WS per worker — within uvicorn's async capability
  • At 6K: Need 3 backend containers (15 workers total, ~400 each) behind load balancer with WebSocket-aware routing
  • No session sharing needed: Chat history lives within the WS connection lifetime only

3.4 Single-Container Architecture High

Current docker-compose runs one backend container with 5 workers. No horizontal scaling, no failover.

  • Scale horizontally: docker compose up --scale backend=3 (15 total workers)
  • Add load balancer: nginx or Traefik with health-check routing and WebSocket support
  • Production: Kubernetes or ECS with autoscaling (target: CPU 70% or WS count per pod)
