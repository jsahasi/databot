Benchmark — REST vs MCP vs DB

ON24 REST API vs MCP Server vs Direct DB

Three-way response-time benchmark and access-pattern recommendations for ON24 Nexus data operations

Version 2.0 Status Reference Last Updated March 2026 Owner Engineering

1. Summary

ON24 Nexus has three ways to access data:

  • ON24 REST API — HTTP calls to the ON24 REST API v2 (PROD: api.on24.com, QA: apiqa.on24.com). Benchmark ran against QA.
  • MCP Server — FastMCP on port 8001 (on24-mcp/server.py, 67 tools) wrapping the REST API with standardized tool-calling protocol for AI agents
  • Direct DB — asyncpg connection to ON24 master PostgreSQL (10.3.7.233:5458/on24master)

For all read operations, direct DB is fastest (10–100× faster than REST). The MCP server adds ~50–150 ms overhead per call on top of REST API latency (session init + HTTP transport). REST API / MCP should be reserved for write operations and endpoints with no DB equivalent.

These numbers are representative estimates. Run python -m tests.benchmark_read_apis from backend/ for a full 3-way comparison of all 50 read tools (requires VPN, MCP server running, and .env.local credentials).

5. Recommendations

Use Direct DB For
  • All read-only analytics queries
  • Attendance trends & comparisons
  • Top events by any metric
  • Audience company breakdowns
  • Poll questions + response counts
  • Engagement KPIs
  • Attendee + registrant lists
  • Event search and filtering
Use MCP Server For
  • AI agent tool calls (standardized protocol)
  • Write operations via agents
  • Endpoints with no DB equivalent (37 tools)
  • Config/contact lookups
  • External integrations needing tool protocol
  • Centralized REST API auth
Use REST API Directly For
  • Backend services bypassing MCP
  • Bulk operations (avoid MCP overhead)
  • Prospect Engagement Profile
  • Non-agent write operations

The key architectural rule: the ON24 master DB (ON24_RO) is read-only by design. All write operations are funneled through the MCP server / REST API via the Admin Agent. Analytics reads use direct DB (10–100× faster). The 37 MCP tools without DB equivalents are the only case where MCP overhead is unavoidable — and for config/helper lookups the overhead is minimal (250–500 ms total).

6. Methodology

Two benchmark scripts are available:

  • backend/tests/benchmark_api_vs_db.py — Original 2-way (REST vs DB) for 10 core analytics operations.
  • backend/tests/benchmark_read_apis.py — Comprehensive 3-way (REST vs MCP vs DB) covering all 50 read tools in 7 phases.

Common methodology:

  • Each operation is executed 3 times; the median is reported to reduce the effect of cold-start and GC pauses.
  • Timing is measured via time.perf_counter() wrapped around the entire async coroutine (including connection acquisition from the pool).
  • A real event_id is resolved from recent past events before per-event benchmarks run.
  • REST API calls use ON24Client with token-bucket rate limiting per endpoint category.
  • MCP calls use streamablehttp_client connecting to localhost:8001/mcp; each call creates a new ClientSession (init + tool call), matching the stateless production behavior.
  • DB queries use asyncpg pool (min_size=1, max_size=5) against on24master on 10.3.7.233:5458 — requires VPN.
  • Exceptions are caught per operation; unavailable endpoints are recorded as errors rather than crashing the benchmark.

9. Architecture Note

The triple-access design is intentional and load-bearing:

  • The ON24 master DB (ON24_RO) is a read-only replica — writes are blocked at the database level, preventing accidental data modification by agents.
  • The REST API enforces ON24's business logic for write operations (confirmation emails, event provisioning, registration workflow) which cannot be replicated by direct INSERT/UPDATE.
  • The MCP server (on24-mcp, FastMCP on port 8001) provides a standardized tool-calling interface for AI agents. It wraps the REST API with stateless HTTP transport — each tool call creates a new ON24Client, so MCP inherits REST API rate limits plus ~50–150 ms transport overhead. 67 tools total: 50 read + 14 write + 3 GDPR.
  • The rate limiter (6-tier token bucket, backend/app/services/rate_limiter.py) protects against REST API quota exhaustion. Direct DB queries bypass this entirely. MCP calls inherit the rate limiter via ON24Client.
  • All agent tool calls are audit-logged to agent_audit_logs regardless of which data source is used.
