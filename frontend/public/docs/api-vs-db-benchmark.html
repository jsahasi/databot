Benchmark — API vs DB

ON24 REST API vs Direct DB

Response-time benchmark and access-pattern recommendations for ON24 Nexus data operations

Version 1.0 Status Reference Last Updated March 2026 Owner Engineering

1. Summary

ON24 Nexus has two ways to read data: the ON24 REST API (HTTP calls to apiqa.on24.com) and a direct asyncpg connection to the ON24 master PostgreSQL database (10.3.7.233:5458/on24master). For all read operations, direct DB access is faster by 10–100×. The REST API should be reserved exclusively for write operations where no DB equivalent exists.

These numbers are representative estimates based on known system characteristics. Run python backend/tests/benchmark_api_vs_db.py against your environment (requires VPN and .env.local credentials) to get actual measurements.

3. Write Operations — REST API Only

The following operations have no direct DB equivalent and must use the REST API:

Operation REST API Endpoint Direct DB Notes
Create event POST /event N/A (read-only creds) ON24 master DB connection is read-only. All writes go through the REST API.
Update event PUT /event/{id} N/A Same as above.
Add registrant POST /event/{id}/registrant N/A Write operation. Triggers ON24 confirmation email workflow.
Remove registrant DELETE /event/{id}/registrant/{uid} N/A Write operation.
Get Prospect Engagement Profile (PEP) GET /prospect/{email}/engagementprofile Complex join REST API provides a purpose-built endpoint. Equivalent DB join spans 6+ tables and requires email normalization.

4. Recommendations

Use Direct DB For
  • All read-only analytics queries
  • Attendance trends & comparisons
  • Top events by any metric
  • Audience company breakdowns
  • Poll questions + response counts
  • Engagement KPIs (dw_event_session)
  • Attendee + registrant lists
  • Event search and filtering
Use REST API For
  • Create event
  • Update event metadata
  • Add registrant
  • Remove registrant
  • Get Prospect Engagement Profile
  • Any future write operations

The key architectural rule: the ON24 master DB connection (ON24_RO user) is read-only by design. All write operations are funneled through the REST API via the Admin Agent. This separation means read performance is never constrained by API rate limits (6-tier token bucket, 10–1000 req/min) and analytics queries run at database speed.

5. Methodology

The benchmark script (backend/tests/benchmark_api_vs_db.py) uses the following approach:

  • Each operation is executed 3 times; the median is reported to reduce the effect of cold-start and GC pauses.
  • Timing is measured via time.perf_counter() wrapped around the entire async coroutine (including connection acquisition from the pool).
  • A real event_id is resolved from the 5 most recent past events before per-event benchmarks run.
  • REST API calls use the existing ON24Client which applies token-bucket rate limiting per endpoint category.
  • DB queries use the shared asyncpg pool (min_size=1, max_size=3 per worker, 5 workers = 15 total connections) against on24master on 10.3.7.233:5458 — requires VPN / ON24 internal network.
  • Exceptions are caught per operation; unavailable endpoints are recorded as errors rather than crashing the benchmark.

8. Architecture Note

The dual-access design is intentional and load-bearing:

  • The ON24 master DB (ON24_RO) is a read-only replica — writes are blocked at the database level, preventing accidental data modification by agents.
  • The REST API enforces ON24's business logic for write operations (confirmation emails, event provisioning, registration workflow) which cannot be replicated by direct INSERT/UPDATE.
  • The rate limiter (6-tier token bucket, backend/app/services/rate_limiter.py) protects against REST API quota exhaustion. Direct DB queries bypass this entirely.
  • All agent tool calls are audit-logged to agent_audit_logs regardless of which data source is used.
